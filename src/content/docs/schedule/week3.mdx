### Software Engineering Topics

#### üîß Implementation

<details>
<summary><strong>Java: JavaDoc</strong></summary>

<div style={{padding: '1rem', background: '#f8f9fa', borderRadius: '8px', marginTop: '1rem'}}>

**JavaDoc** is a documentation generator for Java code.

**Basic syntax:**
```java
/**
 * Returns the sum of two integers.
 * 
 * @param a the first integer
 * @param b the second integer
 * @return the sum of a and b
 */
public int add(int a, int b) {
    return a + b;
}
```

**Key tags:**

@param - describes parameters
@return - describes return value
@throws - describes exceptions
@author - author information
@since - version information

</div>
</details>
<details>
<summary><strong>Java: File I/O</strong></summary>
<div style={{padding: '1rem', background: '#f8f9fa', borderRadius: '8px', marginTop: '1rem'}}>

**Reading files:**
```java
try {
    List<String> lines = Files.readAllLines(Paths.get("data.txt"));
} catch (IOException e) {
    e.printStackTrace();
}
```

**Writing files:**
```java
try {
    Files.write(Paths.get("output.txt"), lines);
} catch (IOException e) {
    e.printStackTrace();
}
```

**Important:** Always handle IOException when doing file operations!
</div>
</details>
<details>
<summary><strong>Java: Packages</strong></summary>
<div style={{padding: '1rem', background: '#f8f9fa', borderRadius: '8px', marginTop: '1rem'}}>

**Packages** organize related classes and interfaces.

**Package declaration:**
```java
package com.mycompany.myapp;
```

**Naming conventions:**

Use lowercase
Use reverse domain name
Separate with dots
e.g., com.example.project.utils

Benefits:

Avoid naming conflicts
Control access
Organize code logically

</div>
</details>
<details>
<summary><strong>Java: JAR Files</strong></summary>
<div style={{padding: '1rem', background: '#f8f9fa', borderRadius: '8px', marginTop: '1rem'}}>

**JAR (Java Archive)** files package Java applications.

**Creating JAR:**
```bash
jar cf myapp.jar -C build/classes .
```

**Creating executable JAR:**

Add manifest with Main-Class
Include all dependencies
Make it runnable with java -jar

</div>
</details>

#### üß™ Testing
<details>
<summary><strong>Developer Testing</strong></summary>
<div style={{padding: '1rem', background: '#f8f9fa', borderRadius: '8px', marginTop: '1rem'}}>

**Developer testing** is testing done by developers themselves.

**Why important:**

Find bugs early
Ensure code works as intended
Confidence in changes
Documentation of behavior

Types:

Unit testing
Integration testing
System testing

</div>
</details>
<details>
<summary><strong>Unit Testing</strong></summary>
<div style={{padding: '1rem', background: '#f8f9fa', borderRadius: '8px', marginTop: '1rem'}}>

**Unit testing** tests individual units/components in isolation.

**Characteristics:**

Test single methods/classes
Fast execution
Isolated from dependencies
Automated

Best practices:

Test one thing at a time
Use descriptive names
Follow AAA pattern (Arrange, Act, Assert)
Keep tests independent

</div>
</details>
<details>
<summary><strong>JUnit</strong></summary>
<div style={{padding: '1rem', background: '#f8f9fa', borderRadius: '8px', marginTop: '1rem'}}>

**JUnit** is a popular Java testing framework.

**Basic test:**
```java
import org.junit.jupiter.api.Test;
import static org.junit.jupiter.api.Assertions.*;

class CalculatorTest {
    @Test
    void testAdd() {
        Calculator calc = new Calculator();
        assertEquals(5, calc.add(2, 3));
    }
}
```

**Common assertions:**

assertEquals(expected, actual)
assertTrue(condition)
assertNull(object)
assertThrows(exception, executable)

</div>
</details>

**Process:**

Fork repository

Make changes
Push to fork
Create PR
Code review
Merge

Clear title and description
Small, focused changes
Include tests
Respond to feedback

</div>

<details>
<summary><strong>Coding Standards</strong></summary>
<div style={{padding: '1rem', background: '#f8f9fa', borderRadius: '8px', marginTop: '1rem'}}>

**Coding standards** ensure consistent, readable code.

**Java conventions:**

Classes: PascalCase (MyClass)
Methods/variables: camelCase (myMethod)

Packages: lowercase (com.example)

Indentation (4 spaces)
Line length (< 120 chars)
Braces on same line
Meaningful names

#### 3. **Coding Standards Exercise** (15 min)

Find coding standard violations in this code:

```java
public class Task
{
    private String description;
    private boolean is_done;
    
    public Task(String d) {
        description=d;
        is_done=false;
    }
    
    public boolean getStatus()
    {
        return is_done;
    }
    
    public void set_done(boolean done)
    {
        is_done = done;
    }
}
```

<details>
<summary>Click to see violations</summary>

Opening brace should be on same line as class/method declaration
Variable name is_done should be camelCase: isDone
Parameter d should have meaningful name: description
Method set_done should be camelCase: setDone
Inconsistent spacing around = operator
Method getStatus() should be named isDone() for boolean

</details>
